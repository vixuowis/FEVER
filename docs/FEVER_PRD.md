# FEVER · 产品与技术规划（重构版）

**当前版本定位**：FEVER 不是“堆很多炫酷面板的终端”，而是一个围绕单条研究任务闭环的金融事件研究工作台。  
**核心目标**：让研究员从“看到事件”走到“形成可复盘的结论”，并把过程沉淀为可追踪、可复用、可校准的研究资产。  
**长期方向**：在研究闭环稳定后，再逐步演进到 Market-Grounded TTRL 自进化引擎。

**Tagline**：Hunt events. Trace echoes.

---

## 一、先纠偏：当前产品真正的问题

### 1.1 不是功能少，而是动作没有产出对象
当前版本已经有事件流、推演树、校准、What-if、历史、模型管理等模块，但多数动作停留在界面状态层：

- 点按钮只是刷新或切换展示，没有形成研究资产
- 写笔记只存在前端内存，没有保存、没有引用、没有复盘价值
- 执行数据能力后只看结果，没有把结果挂到当前研究任务上
- 模块彼此平行摆放，没有围绕一个统一对象协作

### 1.2 我们真正要服务的用户问题
目标用户是中小私募研究员、行业研究员、投研助理。对他们最重要的不是“看到更多卡片”，而是：

1. 快速判断今天最值得研究的事件是什么
2. 明确这次研究到底要回答什么问题
3. 用真实数据拉证据，而不是只看 AI 摘要
4. 形成结构化结论，能给 PM / 同事看
5. 事后能回看这次研究是怎么做的、结论对不对

### 1.3 新的产品判断标准
之后每个按钮、模块、接口都必须回答下面四个问题：

1. 用户此刻要完成什么任务
2. 这个动作会生成什么产出
3. 这个产出会进入下一步哪里被使用
4. 这个产出是否能被保存、复盘、复用

如果回答不了，这个功能就是伪交互。

## 二、产品定位

### 2.1 一句话
FEVER 是一个面向事件驱动研究的工作台，用真实数据、结构化证据和结论沉淀，帮助研究员完成一条完整的研究闭环。

### 2.2 当前阶段定位
- **现在做的是**：Research Terminal / 研究工作台
- **不是现在就要假装做成的**：已经完整落地的自进化 TTRL 引擎
- **当前核心对象**：`Research Case`，也就是一条可持续追踪的研究单

### 2.3 长期定位
当研究工作台的输入、证据、结论、复盘都被结构化记录后，才有条件接入：

- 预测命中评估
- reward 计算
- calibration 更新
- skill / prompt 策略更新
- 长期自进化

也就是说，**研究工作台是 TTRL 的产品地基**，不是和 TTRL 并行的另一条路线。

### 2.4 当前不做什么
- 不做自动交易
- 不做全市场万能终端
- 不做大量彼此独立的展示型面板
- 不做没有保存、没有下游价值的“笔记”“对比”“沙盒”按钮
- 不把“模型管理”做成主路径入口，它应退到配置/运营层

## 三、核心对象：Research Case

### 3.1 为什么必须引入 Research Case
当前最大的问题是没有统一对象。事件、洞察、执行结果、正文、笔记、推演树都漂在页面上，彼此没有归属。

所以当前版本必须围绕一个统一对象组织：

```json
{
  "case_id": "case_20260622_001",
  "event_id": "evt_xxx",
  "title": "贵州茅台 Q2 超预期是否带动白酒板块扩散",
  "status": "in_progress",
  "owner": "王研究员",
  "question": "这次财报超预期是否会带来板块级扩散，哪些标的值得继续跟踪？",
  "scope": {
    "symbols": ["600519.SH", "000858.SZ", "000568.SZ"],
    "horizon": "t+1 ~ t+5"
  },
  "insight_summary": "...",
  "evidence_items": [],
  "hypotheses": [],
  "conclusion": {
    "summary": "",
    "risks": [],
    "next_actions": []
  },
  "timeline": [],
  "created_at": "",
  "updated_at": ""
}
```

### 3.2 Research Case 的核心字段

| 字段 | 作用 | 是否 P0 |
|---|---|---|
| `event_id` | 当前研究绑定哪条事件 | 是 |
| `question` | 这次研究到底要回答什么 | 是 |
| `scope` | 范围、标的、时间窗 | 是 |
| `insight_summary` | 实时洞察的归档结果 | 是 |
| `evidence_items` | 执行能力后拿到的证据集合 | 是 |
| `conclusion` | 最终结论、风险、下一步 | 是 |
| `timeline` | 记录关键动作和时间点 | 是 |
| `what_if_scenarios` | 假设分支 | 否，P1 |
| `review_result` | 事后验证结果 | 否，P1 |
| `reward/calibration` | 自进化反馈 | 否，P2 |

### 3.3 新的产品原则
- 所有动作都必须落到某个 `Research Case`
- 所有模块都必须服务于 Case 的某个阶段
- 用户不是“浏览模块”，而是“推进一条研究单”

## 四、核心用户与高频任务

### 4.1 用户角色

| 角色 | 目标 | 频率 |
|---|---|---|
| 行业研究员 | 对单个事件快速形成研究结论 | 高频 |
| 投研助理 | 帮研究员拉证据、整理正文、沉淀结论 | 高频 |
| PM / 负责人 | 快速看结论、风险、证据充分性 | 中频 |
| 模型/产品运营 | 看调用稳定性、成本、结果有效性 | 低频 |

### 4.2 用户最核心的 3 个任务
1. 从事件池中挑一条最值得研究的事件
2. 围绕明确问题采证并形成结论
3. 事后能回看结论、证据和效果

## 五、重新定义核心用例

### 5.1 用例 A：研究员做一条当天事件研究
角色：消费组研究员  
触发：某白酒龙头财报超预期

```text
1. 在事件池中筛出高优先级事件
2. 进入该事件，创建 Research Case
3. 填写研究问题：
   "这次财报是否会带来板块级扩散？哪些标的值得跟踪？"
4. 刷新实时洞察，得到第一版摘要
5. 执行 1~3 个能力模板：
   - 财报/新闻正文
   - 相关标的表现
   - 可比历史案例
6. 把有效结果保存为 evidence items
7. 形成结论：
   - 主要判断
   - 关键风险
   - 下一步动作
8. 将 Case 标记为已完成或待跟踪
```

**价值**：10 分钟内形成一条可审阅、可复盘、可继续跟踪的研究单。

### 5.2 用例 B：PM 快速审阅一条研究单
角色：基金经理 / 研究负责人

```text
1. 打开某条 Research Case
2. 先看结论和置信状态
3. 再看支撑证据是否足够
4. 判断是否需要追加问题或继续采证
5. 给出下一步动作：继续跟踪 / 补证据 / 暂不处理
```

**价值**：PM 不需要重新读一堆新闻和原文，只看研究产物本身。

### 5.3 用例 C：事后复盘一条研究单
角色：研究员 / 产品运营

```text
1. 打开历史 Case
2. 查看当时的问题、证据、结论
3. 对照后续市场或事件结果
4. 判断这次研究结论是否有效
5. 标记哪些证据最有用、哪些动作是噪音
```

**价值**：开始积累真正可复用的研究经验，而不是只有临时页面状态。

## 六、核心流程：一条研究闭环

### 6.1 P0 必须落地的主流程

```text
发现事件
  ↓
选择事件
  ↓
创建/进入 Research Case
  ↓
定义研究问题与范围
  ↓
刷新实时洞察
  ↓
执行能力模板采证
  ↓
沉淀证据
  ↓
写结论 / 风险 / 下一步
  ↓
保存并进入待跟踪/已完成
```

### 6.2 每一步的输入、动作、产出、下一步

| 阶段 | 用户动作 | 输入 | 产出 | 下一步 |
|---|---|---|---|---|
| 发现事件 | 浏览和筛选事件池 | 事件流 | 候选事件 | 选择事件 |
| 选择事件 | 进入事件详情 | 事件卡片 | 当前事件上下文 | 创建 Case |
| 定义问题 | 填写问题、标的、时间窗 | 事件上下文 | `question + scope` | 刷新洞察 |
| 刷新洞察 | 获取首轮摘要 | 事件 + 问题 | `insight_summary` | 采证 |
| 执行采证 | 执行能力模板 | 问题 + 数据源 | 原始执行结果 | 保存证据 |
| 保存证据 | 选择有效结果 | 执行结果 / 正文 | `evidence_items` | 写结论 |
| 写结论 | 填写判断与风险 | 洞察 + 证据 | `conclusion` | 提交/待跟踪 |
| 复盘 | 对照后续表现 | 历史 Case | `review_result` | 反馈模型/流程 |

### 6.3 当前 UI 必须围绕这条流程重排
- 顶部：当前 Case 状态、下一步、保存状态
- 左侧主区：事件池和候选事件
- 右侧主区：当前 Case 的问题、证据、结论
- 次级模块：推演树、What-if、历史案例都应从属于当前 Case

## 七、模块重构：每个模块到底服务什么

### 7.1 模块与主流程映射

| 模块 | 新职责 | 对应流程阶段 | 是否保留为 P0 |
|---|---|---|---|
| 事件流 | 发现候选事件、创建 Case | 发现事件 / 选择事件 | 是 |
| 研究会话条 | 显示当前 Case 状态与下一步 | 全流程导航 | 是 |
| 实时洞察 | 生成第一版研究摘要 | 刷新洞察 | 是 |
| 能力执行台 | 拉真实证据 | 执行采证 | 是 |
| 证据面板 | 保存和管理 evidence items | 保存证据 | 是 |
| 结论区 | 记录结论/风险/下一步 | 写结论 | 是 |
| 推演树 | 组织假设与证据结构 | 洞察 / 采证 / 结论 | P1 |
| 历史库 | 检索相似 Case 或历史证据 | 采证 / 复盘 | P1 |
| What-if | 生成假设分支并附着到 Case | 分析深化 | P1 |
| 校准面板 | 看历史效果与方法质量 | 复盘 | P1 |
| 模型管理 | 配置层与运营层，不是主研究路径 | 配置 | P2 |

### 7.2 需要降级或移出主路径的模块
- `模型管理` 不该跟事件流同级竞争用户注意力
- `What-if` 在没有保存分支前只是演示功能，先降级为次级分析工具
- `Compare Desk` 如果不能产出差异结论，就不应占主界面核心位置
- `liveMode` 如果不能改变请求策略或视图，不应作为主按钮存在

## 八、P0 功能定义：必须产生真实价值

### 8.1 P0 功能清单

1. **Case 创建与保存**
   - 从事件卡创建研究单
   - 保存问题、范围、状态

2. **洞察归档**
   - 实时洞察不仅展示，还能写入当前 Case

3. **证据沉淀**
   - 执行结果可被挑选、保存为 evidence item
   - 正文可摘录重点并归档到 Case

4. **结论输出**
   - 结论必须包含：判断、风险、下一步
   - 结论保存后可供 PM 快速审阅

5. **Case 视图**
   - 一个页面能看完整链路：问题、洞察、证据、结论、时间线

6. **历史 Case 列表**
   - 能看自己做过哪些研究单
   - 能打开继续编辑或复盘

### 8.2 P0 明确不做
- 不做复杂世界模型叙事
- 不做大而全的终端导航
- 不做仅用于“展示技术感”的图表
- 不做无法保存的高级交互

## 九、数据与接口要求

### 9.1 当前已有能力
- 事件面板：`/api/dashboard`
- 实时洞察：`/api/live/insight`
- 能力执行：`/api/live/qveris/execute`
- 正文加载：`/api/live/qveris/detail`

### 9.2 P0 新增后端接口

| 接口 | 作用 |
|---|---|
| `POST /api/cases` | 创建 Research Case |
| `GET /api/cases` | 获取 Case 列表 |
| `GET /api/cases/{id}` | 获取单条 Case |
| `PATCH /api/cases/{id}` | 更新问题、范围、状态、结论 |
| `POST /api/cases/{id}/evidence` | 保存执行结果或正文摘录为证据 |
| `GET /api/cases/{id}/timeline` | 查看研究动作时间线 |

### 9.3 数据抽象原则
- 模型和数据源都是配置，不在 UI 写死供应商名
- Research Case 是业务主对象
- 执行结果不是临时响应，而是可选地进入 Case 证据库

## 十、成功指标

### 10.1 产品指标
- 从进入事件到形成第一版结论的时间 < 10 分钟
- 每条研究单至少包含 1 个问题、2 条证据、1 段结论
- 用户能在历史中找到并继续编辑旧研究单
- PM 能在 1 分钟内看懂一条研究单结论

### 10.2 质量指标
- 执行结果保存成功率
- 证据被引用进结论的比例
- 研究单完成率
- 历史研究单复用率

### 10.3 未来模型指标
- 预测命中率
- calibration error
- reward 提升
- skill 选择效果

这些属于 P1/P2，不应伪装成当前 P0 已完成能力。

## 十一、路线图

### 11.1 Phase 0：研究闭环先跑通
- 建立 Research Case
- 打通问题-洞察-证据-结论-时间线
- 让所有按钮都能产生保存后的产出

### 11.2 Phase 1：研究资产化
- 历史 Case 检索
- What-if 分支保存
- 推演树和证据双向绑定
- Case 复盘与结果标注

### 11.3 Phase 2：研究引擎化
- calibration 面板真正接预测结果
- review -> reward -> skill 更新
- Market-Grounded TTRL 闭环

---

## 十二、当前结论

FEVER 现阶段最应该做的不是继续横向加模块，而是把产品主线收紧成一条真正有价值的研究闭环：

**事件 -> 问题 -> 洞察 -> 证据 -> 结论 -> 复盘**

如果没有这条闭环，再多按钮也只是“会动的展示”；  
只有当每一步都能产生产物，并且进入下一步被继续使用，FEVER 才会从 demo 变成真正可用的研究工作台。
